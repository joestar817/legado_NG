# AI 扫书 v0 端到端协议

## 目标

第一版用真实模型和真实长篇小说验证以下闭环：

```text
简介/目录/开头/结尾定向初扫
→ 基础判断和重大风险前置
→ 连续章节完整扫描
→ 结构化扫描增量落记忆
→ 多维画像聚合
→ 用户跨多个路径反复下钻
→ 证据不足时仅局部补读
```

本协议是验证版，不代表 12 个维度和术语边界已经最终冻结。评测结果应反向推动字段、维度和提示词调整。

## 成功条件

1. 初扫不读取全书，也能根据简介、目录、开头和结尾给出有用但明确标为临时的基础判断。
2. 重大关系或结局风险一旦在已读范围被发现，就出现在概览，不要求用户沿单一路径深入。
3. 完整扫描只按连续章节推进，覆盖集合由实际完整读取结果计算，不能用最大章节号冒充全书覆盖。
4. 每批结果保留事实、事件、维度信号、证据章节和未决项，不只保存一段总结。
5. 后续切换感情关系、主角体验、剧情逻辑、节奏和结局等路径时，可由记忆直接回答，不重新读取已覆盖正文。
6. 未完整覆盖或作品未完结时，只能使用“已发现/暂未发现/疑似/持续中”等状态。

## `book_scan_delta`

模型每次读完新正文后输出一个隐藏 JSON 块。App 只接受与本轮 MCP 实际完整读取章节相交的 `observed_chapters`，再写入：

- `manifest`：总章节数、完整覆盖集合、维度聚合、事件索引和未决项；
- `shard`：本轮章节集合、摘要和原始维度信号；
- `event`：可按术语和参与者检索的关键事件。

正文窗口返回某章 `included_chars < content_chars` 或 `truncated_by_mcp=true` 时，该章不得计入覆盖率。

模型偶尔会遗漏隐藏块。为避免已经读取的正文和 Token 白白浪费，App 还有一条受限兜底：仅在 `book_scan` Skill 激活且 MCP 确认完整读到新章节时，自动保存真实覆盖集合和本轮可见报告；不会凭模型自报补齐章节，也不会调用通用 `agent_memory_upsert`。后续模型重新输出规范增量时，维度信号和事件会继续合并进同一 manifest。

正文 MCP 在发给模型前会移除缓存内容中的内嵌 `data:image/*;base64,...` 图片标签，原始缓存文件不变。返回值同时保留 `raw_content_chars`、清洗后的 `content_chars`、`removed_non_text_chars` 和截断标记。

每次扫书回合落库后，下一轮模型上下文会移除正文工具结果、工具调用中间消息、隐藏协议块和 reasoning，只保留系统规则、书籍上下文与可见对话。完整聊天记录仍保存在 UI 历史中；多维下钻优先从 `book_scan` 记忆读取，不通过保留整段正文来“假装复用”。

## 首轮评测书

- `天之下.txt`：用于检验复杂群像、关系状态变化、强迫/婚恋/亲密边界、未完结状态和“项目术语定义覆盖模型常识”。
- `我上讲台念情书，高冷校花后悔了 作者：狐菌.txt`：用于检验简介已明确路线、后宫/多女主正向属性、关系收束、爽文主角体验和完结结局。

二者不是简单“坏书/好书”或“有雷/无雷”的二元标签，而是用来验证同一事实协议能否生成不同画像。

## 当前端到端入口

1. 在书籍详情页进入 `AI 扫书`。
2. App 附带当前书籍上下文并加载内置 `book_scan` Skill。
3. 初扫自动执行并返回首屏报告。
4. 用户点击“继续扫描”，Agent 从记忆中第一个覆盖缺口继续。
5. 用户可以随时点击多个维度，验证是否复用已有扫描记忆。

评测脚本 `scripts/book_scan_eval.py` 使用相同 Skill 和增量协议，可在不依赖 App UI 的情况下调用 OpenAI-compatible API，生成可恢复的本地档案。

## 2026-07-11 首次真实闭环结果

- 两本本地 TXT 均已通过评测脚本完成目录解析、初扫和至少一个连续正文批次；默认脚本仍指向硅基流动，因当前环境没有 `SILICONFLOW_API_KEY`，本次 live 协议验证使用 App 已配置的 OpenAI-compatible DeepSeek 模型。
- 《天之下》验证出项目口径下的强迫婚姻可同时映射绿帽和送女，并正确保持连载状态；《我上讲台念情书，高冷校花后悔了》验证出后宫、全收倾向和 HE 收束。
- 真机从书籍详情点击 `AI 扫书` 后，自动加载当前书、内置 Skill 和只读 MCP 能力。
- 《我上讲台念情书，高冷校花后悔了》真实初扫经 App 校验覆盖 18/748 章，生成 manifest 和带 1532 字报告的 shard。
- 点击“感情关系”只调用一次 `agent_memory_search`，没有读取正文；随后完整扫描从首个缺口继续，累计覆盖到 57/748 章，并写入 5 组维度信号和 5 个事件。
- 扫书上下文清理后，持久化上传上下文从包含大量正文工具结果的 25 条消息降为 4～6 条、约 1.2 万字符。

v0 仍允许模型漏掉结构化增量，此时只有覆盖集合和可见报告是权威持久化结果；这是为了保证首版可用的降级路径，不代表结构化事件提取已经完成最终定型。
