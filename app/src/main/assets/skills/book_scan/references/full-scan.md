# 扩展排雷扫描

只在用户明确选择继续排雷或指定扫描范围后加载。继续使用专用入口 App 上下文固定的原样 `work_key`；不得跟随全局当前在读书籍。

v2.5 的继续扫描不再做“目录/snippet 导航”。用户选择的是下一轮要补多少连续正文，不是让模型到处挑疑点。这样牺牲一点灵活性，换取可理解的进度、稳定的覆盖和更少的工具调用。

## 扫描目标

- `scan_100`：从最早未完整读过的章节缺口开始，连续补读约 100 章。
- `scan_300`：从最早未完整读过的章节缺口开始，连续补读约 300 章。
- `scan_all_chapters`：尽量补齐当前目录里所有尚未完整读过的章节。
- `custom_scan_target`：按用户输入的章号或范围补读。

如果已有 manifest 明确保存了下一轮建议范围，且它仍然是连续缺口，可以从该范围开始；否则从最早的完整正文缺口开始。不要为了某一本书或某个角色临时改扫描策略。

## 覆盖口径

每轮扫描只认一种可用于结论的覆盖：完整正文覆盖。

manifest 的 `data` 至少保留：

- `total_chapters`：当前目录总章数；
- `full_text_coverage`：已经完整读取正文的 0-based 半开区间数组，例如 `[[0,10],[405,410]]`；
- `next_scan_ranges`：下一轮建议补读的 0-based 半开区间数组。

更新后合并重叠区间。不要把目录浏览、章节标题或片段浏览混进 `full_text_coverage`。旧档案如果只有片段或模糊记录，只能作为历史参考，不能当成完整正文覆盖。

这些字段是给宿主计算进度用的。报告和交互说明里不要出现 `manifest`、`full_text_coverage`、`scan_100`、`snippet`、`ch257-405` 这类内部词。

## 执行顺序

1. 查询唯一 manifest 和章节目录，必要时带 `include_cache_status=true`。
2. 根据 `full_text_coverage` 算出当前目录里尚未完整读过的连续缺口。
3. 选择本轮目标：
   - `scan_100`：取第一个缺口内最多 100 章；
   - `scan_300`：取第一个缺口内最多 300 章；
   - `scan_all_chapters`：按缺口顺序尽量补齐；
   - 自定义范围：只读用户指定范围。
4. 用 `bookshelf_text_window_get` 连续读取正文。每次窗口建议 10～20 章，`char_limit=0`。如果返回截断，缩小窗口重试，不要把截断当完成。
5. 每批保存一次 window bundle / 扫描事实，并更新同一 manifest。顶层 `source_receipt_ids` 必须包含本批正文回执。
6. 目标未完成时继续下一批。批次之间不加载报告、不输出进度文案、不让用户重新选择。

## 停止条件

只有这些情况可以停止：

- 用户取消；
- 正文不可用或缓存缺失；
- 连续两次缩小窗口仍失败；
- Provider 或工具调用真实失败；
- 已完成用户选择的本轮范围；
- 上下文已接近极限，必须先保存再交回。

停止前先保存已完成内容和 manifest。结束说明只写用户能看懂的话：

- “本轮完整读了第 X—Y 章。”
- “目前完整读过第 A—B 章、第 C—D 章。”
- “第 M—N 章还没完整读，不能排除隐藏雷点。”

目标完成、用户停止或明确要求分析时，才加载 `book_scan_report`，综合已保存事实与本轮最新正文结果，输出一次变化报告。
